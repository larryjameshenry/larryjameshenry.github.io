Subscription Vending: Automating New Workload Onboarding with IaC

Apr 8, 2026 min read

Cloud adoption at scale fails when the platform team is a manual bottleneck. If onboarding a new application team requires a Jira ticket, three emails, and 10 business days of manual configuration, you haven’t built a platform—you’ve built a bureaucracy. In an enterprise landing zone, we treat the Subscription as a product. We use a Subscription Vending pattern to automate the creation and configuration of every new environment through a pull request.

Subscription Vending ensures that every new subscription is “Secure by Default” and “Connected by Design.” When the vending machine runs, it doesn’t just create a billing entry; it places the subscription in the correct Management Group, peers its VNet to the hub, assigns mandatory RBAC roles, and establishes a budget. This guide teaches you how to implement a PR-driven vending workflow using Azure Verified Modules (AVM) and establish a self-service culture for your engineering teams.

1. The PR-Based Vending Workflow

The goal of vending is to move the conversation from “Can you create this for me?” to “Here is the code I want to run.”

PR-Based Vending Workflow

By defining your workloads in a simple YAML or JSON file in a Git repository, you gain a full audit trail of who requested what and when. The platform team merely reviews and approves the PR, letting the automation handle the rest.

2. The Subscription Vending Machine

The vending machine is a modular piece of IaC that coordinates six distinct deployment steps.

The Subscription Vending Machine Architecture

2.1: Creating the Subscription Alias

In 2026, we use the Microsoft.Subscription/aliases resource to provision subscriptions programmatically. This resource is idempotent; if you run it again with the same name, it returns the existing subscription rather than creating a duplicate.

Bicep Pattern:

targetScope = 'tenant'

module subVend 'br/public:avm/ptn/lz/sub-vending:0.5.0' = {
  name: 'subVend-${workloadName}'
  params: {
    subscriptionAliasEnabled: true
    subscriptionBillingScope: '/providers/Microsoft.Billing/billingAccounts/12345/enrollmentAccounts/67890'
    subscriptionDisplayName: workloadName
    subscriptionWorkload: 'Production'
  }
}

3. Automation Patterns: The “90-Second Rule”

A common failure in subscription automation is the Identity Propagation Delay. When you create a new subscription, it takes approximately 60-90 seconds for Entra ID to recognize the new scope. If your IaC immediately attempts to assign a role, it will fail with a PrincipalNotFound or ScopeNotFound error.

Terraform Fix: Use the time_sleep resource to introduce a mandatory pause after subscription creation.

resource "time_sleep" "wait_for_subscription" {
  create_duration = "90s"
  depends_on      = [azurerm_subscription.workload]
}

4. Self-Service IPAM: Avoiding Overlaps

The hardest part of scaling subscription vending is IP Address Management (IPAM). If two teams request the same 10.1.1.0/24 range, your VNet peering will fail.

In 2026, we include a CIDR Overlap Check in our CI/CD pipeline. A Python script reads a central ipam.yaml file in the repo, validates the requested range doesn’t overlap with any existing allocations, and blocks the PR if a collision is detected. This allows teams to choose their own IPs while the platform enforces the global network plan.

5. Security and Billing Guardrails

Vending is the perfect time to apply mandatory controls. Every subscription should receive:

  • RBAC: An “App Admin” group for the workload team.
  • Budget: A $500/month starting budget with automated alerts to the platform team.
  • Diagnostics: A DINE policy (from Article 4) that automatically connects the subscription to the hub’s Log Analytics Workspace.

Key Takeaways

  1. PR-Driven is GitOps: Use Git as the source of truth for your entire cloud footprint.
  2. Idempotency is Safety: Use aliases and AVM pattern modules to ensure your automation can run repeatedly without errors.
  3. Mind the Propagation: Include a 90-second wait after creation to allow the Azure control plane to synchronize.
  4. Enforce CIDR Integrity: Shift IPAM validation into the PR phase to avoid unpeered “Ghost Spokes.”
