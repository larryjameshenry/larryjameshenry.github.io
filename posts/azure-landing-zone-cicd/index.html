CI/CD Pipeline for Your Landing Zone: Deploying Azure Verified Modules with GitHub Actions

Apr 8, 2026 min read

Deploying an enterprise landing zone from a local terminal is a single point of failure. It lacks an audit trail, bypasses peer review, and relies on long-lived administrative secrets stored on personal machines. A production-grade foundation requires a GitOps approach, where every change to the infrastructure is managed as code, tested in a pipeline, and deployed via a secure, secret-less automation identity.

The Landing Zone CI/CD Pipeline transforms your foundation from a collection of scripts into a reliable, repeatable platform service. In 2026, the standard for these pipelines is Workload Identity Federation (OIDC), which eliminates static API keys, and Shift-Left Testing using tools like PSRule. This guide teaches you how to build a production pipeline using GitHub Actions to deploy Azure Verified Modules (AVM) and establish a secure deployment perimeter for your landing zone.

1. The PR-Driven Plan-First Workflow

The goal of the pipeline is to ensure that no change reaches Azure without a human review of the predicted impact. We implement a Plan-First workflow.

PR-Driven Plan-First Workflow

When a developer opens a PR, the pipeline runs a “What-If” analysis and posts the result as a comment. This allows the reviewer to see exactly which resources will be created, modified, or destroyed before clicking merge.

2. Secure Authentication with OIDC

Storing AZURE_CLIENT_SECRET in GitHub is a high-risk legacy pattern. If the secret is leaked, an attacker has permanent access. OpenID Connect (OIDC) solves this by using a trust relationship between GitHub and Entra ID.

GitHub OIDC Security Boundary

In 2026, we use Environment-scoped Federated Credentials. This ensures that the production deployment identity can only be used by a workflow running in the production environment branch on GitHub, providing a cryptographic boundary between your dev and prod infrastructure.

3. Shift-Left Testing with PSRule

Compliance should be caught in the pipeline, not in a post-deployment audit. We use PSRule for Azure to validate our Bicep and Terraform code against the Well-Architected Framework (WAF) before it is deployed.

# GitHub Actions: PSRule WAF Compliance
- name: PSRule WAF Compliance
  uses: microsoft/ps-rule@v2
  with:
    modules: PSRule.Rules.Azure
    inputPath: live/

If a developer attempts to deploy a VNet without a diagnostic setting or an unencrypted Storage Account, PSRule fails the build, preventing the non-compliant code from ever reaching the main branch.

4. Safe State Storage Architecture

Your Terraform state files or Bicep Deployment Stacks contain the blueprint of your entire cloud estate. They must be protected as high-value assets.

Safe State Storage Architecture

By using Private Endpoints and RBAC-only access (disabling storage account keys), you ensure that only your authorized CI/CD runners can read or write the state.

5. Bicep Deployment Stacks: The Native Way

For Bicep users, Deployment Stacks (GA in 2025) provide the state management and “Action on Unmanage” features previously only found in Terraform.

az stack mg create \
  --name "lz-management-layer" \
  --management-group-id "mg-platform" \
  --deny-settings-mode "denyDelete" \
  --action-on-unmanage "deleteResources"

The --deny-settings-mode denyDelete flag acts as a native resource lock, preventing anyone (including owners) from manually deleting resources managed by the stack.

Key Takeaways

  1. Peer Review is Mandatory: Use PR-driven workflows to ensure every platform change has a second pair of eyes.
  2. Go Secret-less: Use OIDC for all automation identities. Treat static client secrets as technical debt.
  3. Validate early: Use PSRule to catch WAF violations in the pipeline rather than the portal.
  4. Protect the State: Isolate your state storage using Private Link and RBAC to prevent tampering.
