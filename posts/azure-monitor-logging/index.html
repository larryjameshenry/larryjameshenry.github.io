Centralized Monitoring: Log Analytics, Diagnostic Settings, and Azure Monitor Workbooks

Apr 8, 2026 min read

If you can’t see it, you can’t govern it. In an enterprise landing zone, observability is not an afterthought—it is the foundation of security and cost management. A fragmented logging strategy, where every application team manages its own Log Analytics Workspace, leads to “Siloed Observability.” Security teams can’t correlate events across VNets, and finance teams can’t identify the source of a cost spike until the bill arrives.

The Centralized Logging Architecture moves all telemetry—resource logs, activity logs, and network traffic—into a single “Hub” workspace owned by the platform team. This guide teaches you how to design and deploy this hub using Azure Verified Modules (AVM) and establish a self-healing logging baseline using Azure Policy and Data Collection Rules (DCRs).

1. Centralized Logging Architecture

The core design principle is the Hub-and-Spoke Telemetry Flow. While applications run in spokes, their diagnostic “signals” are streamed back to the management subscription.

Centralized Logging Architecture

This centralization enables cross-subscription KQL queries, allowing you to answer questions like “Show me all failed login attempts across the entire production estate” in sub-second time.

2. Deploying the Hub with Terraform AVM

Using AVM ensures your workspace follows best practices for retention and tiering from day one.

module "log_analytics_workspace" {
  source  = "Azure/avm-res-operationalinsights-workspace/azurerm"
  version = "~> 0.4"

  name                = "law-prod-mgt-001"
  resource_group_name = "rg-prod-mgt-001"
  location            = "eastus"
  
  # Standard production settings
  sku               = "PerGB2018"
  retention_in_days = 90
  daily_quota_gb    = 50 # Prevent runaway ingestion costs
}

3. Data Collection Rules (DCR) and Cost Optimization

Logging every packet passing through a firewall is expensive. In 2026, we use Data Collection Rules (DCRs) to filter and transform logs at the ingestion point, before they hit your bill.

DCR Ingestion Filtering (Cost Optimization)

By applying a KQL transform to your firewall logs (e.g., dropping ALLOW logs for internal-to-internal traffic), you can reduce your ingestion costs by 40-60% without losing security visibility into high-risk external traffic.

4. Automating Diagnostics with Azure Policy

You cannot rely on developers to remember to enable diagnostic settings. Instead, use a DeployIfNotExists (DINE) policy (from Article 4) to automatically connect every new VNet, Key Vault, and Storage Account to your central hub.

// Example: Policy existence condition for VNet Diag
{
  "field": "Microsoft.Insights/diagnosticSettings/logs.enabled",
  "equals": "true"
}

5. Visualizing the Platform with Workbooks

Data in Log Analytics is only useful if it’s interpretable. Azure Monitor Workbooks provide an interactive canvas for visualizing your KQL queries. A standard “Platform Health” workbook should include:

  • Top 10 Log Producers: Identify noisy resources that need DCR filtering.
  • RBAC Change Audit: Track every permission change across the management group.
  • Firewall Denials: Map outbound blocks to specific application spokes.

KQL Pro Tip: Use parse_json(tostring(...)) to extract deep metadata from the Properties field in AzureActivity logs, as they are often double-encoded.

Key Takeaways

  1. Hub is the Hub: One Log Analytics Workspace for the entire landing zone is the most efficient design for correlation and cost.
  2. Filter at Ingestion: Use DCRs to drop low-value logs and keep your data clean.
  3. Policy is the Glue: Use DINE policies to ensure 100% logging coverage across new subscriptions.
  4. Workbooks over Dashboards: Use Workbooks for deep, interactive investigation and Dashboards for high-level monitoring.
