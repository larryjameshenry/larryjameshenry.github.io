Enterprise Azure Landing Zone: The Complete Guide

Apr 7, 2026 min read

Most Azure environments fail because nobody built a foundation first. One team creates a resource group called test-rg that becomes production 18 months later. Another ships without tags and cannot attribute 40% of the monthly bill. These aren’t isolated mistakes — they are the predictable outcome of deploying workloads before designing the platform that hosts them.

An Azure Landing Zone establishes a governed, scalable platform before workloads arrive. This guide is the pillar of a 10-part series. It covers the architecture and the decisions required to build an enterprise-grade foundation from scratch.

By the end of this guide, you will:

  • Implement the eight CAF design areas.
  • Understand the separation between platform and application landing zones.
  • Choose between Terraform AVM, Bicep AVM, and the ALZ Accelerator.
  • Deploy a production scaffold using modern IaC patterns.

What Is an Azure Landing Zone?

A landing zone is not a product — it is a set of design decisions, encoded in IaC and policy, applied before workloads arrive. It ensures every team inherits the same baseline: hub networking with centralized egress, identity constraints, mandatory tagging with budget alerts, and centralized logging.

The separation is explicit:

  • Platform Landing Zones: Shared services (Hub VNet, Firewall, Bastion, DNS, Logging) managed by the platform team.
  • Application Landing Zones: Spoke subscriptions where product teams deploy application resources.

The Eight CAF Design Areas

Each area addresses a specific failure mode. Skipping one creates a technical debt that is expensive to remediate.

  1. Billing and Entra Tenant: Determining APIs for subscription vending.
  2. Identity and Access: Implementing OIDC secret-less pipelines and PIM.
  3. Management Group Organization: Designing the hierarchy for policy inheritance.
  4. Network Topology: Hub-and-spoke with forced tunneling and IPAM.
  5. Security: Defender for Cloud and centralized Sentinel.
  6. Governance: Enforcing standards via Azure Policy (Audit/Deny/DINE).
  7. Management and Monitoring: Centralized Log Analytics and Workbooks.
  8. Platform Automation: CI/CD pipelines and automated subscription vending.

IaC Tooling: Terraform AVM vs. Bicep AVM

Modern landing zones utilize Azure Verified Modules (AVM). These are Microsoft-maintained modules that follow Azure’s recommended practices.

Terraform AVM

Best for teams already using Terraform or multi-cloud environments. Requires an Azure Storage state backend. Legacy Note: The terraform-azurerm-caf-enterprise-scale module is archived as of August 2026.

Bicep AVM

Native ARM integration with no state file to manage. Uses Deployment Stacks (GA since May 2024) for lifecycle management. Legacy Note: ALZ-Bicep was deprecated in February 2026.

Hands-On: Deploying the Scaffold

Use the ALZ Accelerator to generate your initial repository structure.

Step 1: Install the ALZ Module

Install-Module -Name ALZ -Force -Scope CurrentUser
# 2026 release is 7.0.1+; review breaking changes if upgrading from 6.x

Step 2: Run the Bootstrap

New-ALZEnvironment -Path "C:\Source\MyALZ" -DeploymentStrategy "GitHubActions"

Step 3: Deploy the Hierarchy (Terraform)

module "management_groups" {
  source  = "Azure/avm-ptn-alz/azurerm"
  version = "0.19.0"
  management_group_name = "contoso"
  enable_telemetry      = false
}

Best Practices

  • Hierarchy First: Design your management group tree before writing any networking or application code.
  • IP Planning: Plan address space for three to five years. Resizing VNets after peering is established triggers outages.
  • OIDC Only: Never use client secrets in pipelines. Use OIDC federated identity for all Azure connections.
  • Policy at Scale: Assign policies at the management group level, not the subscription level.
