Hub-and-Spoke Networking for Azure Landing Zones: Azure Firewall, Bastion, and Private DNS

Apr 7, 2026 min read

Networking is the nervous system of your landing zone. In an enterprise cloud environment, you cannot allow every application to manage its own internet egress or hybrid connectivity. Ad-hoc networking leads to overlapping IP ranges, uninspected traffic, and a fragmented DNS namespace that makes cross-service communication impossible.

The Hub-and-Spoke topology is the mandatory standard for enterprise Azure deployments. It centralizes shared services—such as the Azure Firewall, VPN/ExpressRoute Gateways, and DNS Resolvers—into a single “Hub” VNet. All application workloads reside in “Spoke” VNets, which peer back to the hub to share its connectivity while remaining isolated from one another. This guide teaches you how to build the networking core of your landing zone using Azure Verified Modules (AVM) and establish a secure, scalable perimeter for your cloud workloads.

1. The Hub Networking Stack

The hub VNet is the high-security gateway for your entire Azure region. In 2026, the retirement of “Default Outbound Internet Access” means that a hub with a centralized firewall is no longer just a best practice—it is a requirement for resource connectivity.

The Hub Networking Stack

1.1: Mandatory Reserved Subnets

Azure enforces strict naming and sizing rules for hub services. All of these should be at least a /26 to accommodate future scaling:

  • AzureFirewallSubnet: The data plane for the firewall.
  • AzureFirewallManagementSubnet: Required only if you enable “Forced Tunneling” to route management traffic separately.
  • GatewaySubnet: For VPN or ExpressRoute gateways.
  • AzureBastionSubnet: For the Bastion managed service.

2. Deploying the Hub with Terraform AVM

Using AVM modules reduces the 500 lines of standard VNet/Firewall code into a few atomic blocks.

module "hub_vnet" {
  source  = "Azure/avm-res-network-virtualnetwork/azurerm"
  version = "~> 0.7"

  name                = "vnet-prod-hub-001"
  resource_group_name = "rg-prod-conn-001"
  location            = "eastus"
  address_space       = ["10.0.0.0/16"]

  subnets = {
    AzureFirewallSubnet = { address_prefixes = ["10.0.0.0/26"] }
    GatewaySubnet       = { address_prefixes = ["10.0.0.64/27"] }
    AzureBastionSubnet  = { address_prefixes = ["10.0.0.96/26"] }
  }
}

module "firewall" {
  source  = "Azure/avm-res-network-azurefirewall/azurerm"
  version = "~> 0.5"

  name                = "fw-prod-hub-001"
  resource_group_name = "rg-prod-conn-001"
  location            = "eastus"
  sku_tier            = "Premium" # Required for IDPS and TLS inspection
  
  # Link to the hub subnet
  virtual_network_resource_id = module.hub_vnet.resource_id
}

3. Private DNS Resolution Flow

DNS is the most frequent source of post-deployment support tickets. In a hub-and-spoke model, we centralize Private DNS Zones in the hub and use the Azure DNS Private Resolver to handle queries from both cloud and on-premises clients.

Private DNS Resolution Flow (Resolver Pattern)

By linking your hub VNet to the Private DNS Zones, and configuring your spokes to use the Hub’s Firewall or Resolver as their DNS server, you ensure that resource.privatelink.blob.core.windows.net resolves to a private IP across your entire organization.

4. VNet Peering and Gateway Transit

To connect a spoke to the hub, you must create a two-way peering. In an enterprise landing zone, we enable Gateway Transit. This allows spoke VNets to use the VPN or ExpressRoute gateway sitting in the hub, eliminating the need for expensive gateways in every application subscription.

resource "azurerm_virtual_network_peering" "hub_to_spoke" {
  name                      = "peer-hub-to-app01"
  resource_group_name       = module.hub_vnet.resource_group_name
  virtual_network_name      = module.hub_vnet.name
  remote_virtual_network_id = var.spoke_vnet_id
  allow_gateway_transit     = true
}

5. Controlling Egress with User Defined Routes (UDR)

Merely peering a VNet doesn’t force traffic through the firewall. You must apply a Route Table to your spoke subnets with a default route (0.0.0.0/0) pointing to the Azure Firewall’s private IP. This ensures that every packet leaving the spoke is inspected by your central security policy.

Key Takeaways

  1. Centralize Egress: Use Azure Firewall Premium in the hub to inspect all outbound traffic and enforce FQDN filtering.
  2. Size for Growth: Use /26 for mandatory hub subnets to avoid future downtime during service scaling.
  3. DNS is Global: Use the DNS Private Resolver to provide a unified namespace for cloud and on-premises developers.
  4. Automate Peerings: Include peering and UDR configuration in your Subscription Vending process to ensure new spokes are “Secure by Default.”
