Governance at Scale: Writing and Deploying Azure Policies with Terraform and Bicep

Apr 8, 2026 min read

In an enterprise landing zone, configuration drift is the primary enemy. You can build a perfect hub-and-spoke network, but if a developer can manually create a Storage Account with a public endpoint or skip diagnostic logging, your governance model has failed. Azure Policy is the enforcement mechanism that turns your security standards from “recommendations” into “executable logic.”

Azure Policy allows you to govern 1,000 subscriptions with the same effort as one. By assigning policies at the Management Group scope, you ensure that every resource created—today or three years from now—meets your organizational standards. This guide teaches you how to author and deploy policies using Policy as Code patterns in Terraform and Bicep, moving beyond simple auditing to proactive enforcement and automated remediation.

1. The Policy Assignment Hierarchy

Effective governance requires grouping related controls into logical units called Initiatives (or Policy Sets). This allows you to assign a single “Security Baseline” to your hierarchy rather than managing 50 individual assignments.

Policy Assignment Hierarchy

2. Proactive Enforcement with the Deny Effect

The most powerful tool in your governance arsenal is the Deny effect. It evaluates a resource request before it is created. If the resource doesn’t meet the requirement (e.g., it lacks a mandatory tag), ARM rejects the request with a 403 Forbidden.

{
  "if": {
    "field": "Microsoft.Storage/storageAccounts/publicNetworkAccess",
    "ne": "Disabled"
  },
  "then": {
    "effect": "Deny"
  }
}

Pro Tip: Always set the mode property in your policy definition. Use Indexed for policies that evaluate tags and locations. Use All for resource-specific properties like network rules or subnets.

3. Self-Healing with DeployIfNotExists (DINE)

Auditing tells you what’s broken; DeployIfNotExists (DINE) fixes it automatically. This effect is used for auxiliary resources like diagnostic settings or private DNS zones. If the primary resource (e.g., a VNet) exists but the dependent resource (e.g., diagnostic settings) is missing, Azure Policy triggers a remediation task to deploy it.

DINE Policy Remediation Workflow

Terraform Implementation: To support DINE, the policy assignment requires a Managed Identity and an RBAC role assignment allowing that identity to deploy resources.

resource "azurerm_management_group_policy_assignment" "logging" {
  name                 = "deploy-diag-settings"
  management_group_id  = var.platform_mg_id
  policy_definition_id = azurerm_policy_definition.dine_logging.id
  location             = "eastus"
  identity {
    type = "SystemAssigned"
  }
}

resource "azurerm_role_assignment" "remediation_access" {
  scope                = var.platform_mg_id
  role_definition_name = "Contributor"
  principal_id         = azurerm_management_group_policy_assignment.logging.identity[0].principal_id
}

4. Policy as Code: Bicep and loadJsonContent()

Maintaining large JSON policy definitions inside Bicep files is difficult. In 2026, we use the loadJsonContent() function to keep our logic in standalone JSON files while managing the deployment in Bicep.

resource secureStoragePolicy 'Microsoft.Authorization/policyDefinitions@2021-06-01' = {
  name: 'deny-public-storage'
  properties: {
    policyRule: loadJsonContent('policies/storage/deny-public-access.json')
    parameters: loadJsonContent('policies/storage/parameters.json')
  }
}

5. Managing Exceptions with Exemptions

Not every application can meet every policy on Day 1. Exemptions allow you to grant temporary waivers for specific resources or subscriptions without loosening the global standard. Unlike “Exclusions” (which are permanent), Exemptions have an expiration date and a categorized reason (Waiver or Mitigated), providing the audit trail required for compliance reviews.

Key Takeaways

  1. Initiatives over Policies: Group your controls to simplify assignment management at scale.
  2. Deny is Prevention: Use Deny for security boundaries (Public IPs, Open RDP) to stop risks before they exist.
  3. DINE is Maintenance: Use DeployIfNotExists for standard platform boilerplate like logging and backup.
  4. Remediate First: Never switch an existing policy to Deny without running a remediation task to fix existing resources first.
