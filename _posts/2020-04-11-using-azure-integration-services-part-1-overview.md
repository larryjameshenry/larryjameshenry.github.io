I've been working on transition my employeers integrations from the old scheduled pull from the data-tier to an event driving cloud first integration.  There are many great cloud integration software solutions.  Since I focus on Microsoft technologies, I have been designing the new integration using [Azure Integration Services](https://azure.microsoft.com/en-us/product-categories/integration/).  For those curious during the last Ignite, Microsoft scooped several of their technologies and branded them Azure Integration Services.  Those technologies include, Azure API Management, Service Bus, Logic Apps and Event grid.  You are not limited to just these services when designing cloud integration, but they are a really good set of tools to have in your toolbox.

![Azure Integration Diagram](/assets/images/posts/2020-04-11-using-azure-integration-services-part-1-overview/azure-integration-diagram.jpg "Azure Integration Diagram")

[Azure API Management](https://azure.microsoft.com/en-us/services/api-management/) like it's name is an enterprise ready API management solution to secure and separate the concerns of developers, service providers, and consumers.  I find it great for providing integration endpoints for webhooks and with it's connection to [Application Insights](https://docs.microsoft.com/en-us/azure/azure-monitor/app/app-insights-overview) it provides all the telemetry you need to monitor the health and security of you endpoints.  [Azure Service Bus](https://azure.microsoft.com/en-us/services/service-bus/) is the durable messaging component that allows for store and forward or publish and subscribe messaging patterns.  As well as, many other messaging features that you would expect in a mature product.  Skipping ahead, [Event Grid](https://azure.microsoft.com/en-us/services/event-grid/) takes us out of the dark ages and the inevitable poll for data.  Event Grid allows for own processes to be invoked when events fire and without any additional logic in our processes.  We just have to subscribe to certain events in Event Grid and our processes will be triggered when that event happens.  Last but not least is [Logic Apps](https://azure.microsoft.com/en-us/services/logic-apps/), it's the workflow automation that brings us all together.  With hundreds of connectors, workflow logic and built in expressions (a.k.a. Excel like functions) you can build some very nice integrations with a mulitude of systems and services.


For the integration I worked on, the vendor's website wanted me to poll their API for transactions that occurred in their system.  I really didn't like wasting time polling for data, so I talked them into sending a durable message to our webhook when they successfully stored the data. When I say durable, I mean their system would store and forward the message until their system ensured our integration successfully received the message.  Our webhook was created in Azure API Management and secured by a subscription key where the only API assigned was this webhook.  I had different options for the API, but the simplest option was a Logic App listening to an HTTP trigger and storing the HTTP body into an Azure Service Bus Topic.

![Azure Integration part 1](/assets/images/posts/2020-04-11-using-azure-integration-services-part-1-overview/azure-integration-part-1.jpg "Azure Integration part 1")

I chose an Azure Service Bus Topic because I had a couple of processes that wanted a copy of the message when the webhook was invoked.  If there was only one message subscriber, I could of used an Azure Storage storage queue.  The particular subscriber for this integration is another Logic App that will trigger on a message added to the topic and store it in an Azure Service Bus queue.  I called this queue, "myprocess"-working-queue.  I used another Logic App to process the message.  This Logic App was quite a bit larger than the other Logic Apps in this solution.  This is where all the works happens.

![Azure Integration Logic App](/assets/images/posts/2020-04-11-using-azure-integration-services-part-1-overview/azure-integration-logic-app.jpg "Azure Integration Logic App")

Actually processing the message takes several steps and involves multiple records added or updated in our on-premise system.  Since our on-premise CRM is not available as a connector, this made me search for another Azure resource to perfrom the needed functions.  Funny enough it was Azure Functions that I landed on.  I needed a way to run c# and access a few .NET assemblies.  I also like the serverless benefits and the available consumption plan to pay only when the functions run.  I broke out each interaction with our CRM into individual calls, so the Logic App could call an Azure Function, perform a small task, and return allowing the Logic App to detemine the next step.  This removed any complex logic from the Logic App, but allowed it to control and report the entire process's outcome.

![Azure Integration part 2](/assets/images/posts/2020-04-11-using-azure-integration-services-part-1-overview/azure-integration-part-2.jpg "Azure Integration part 2")

As we all know, every good process needs to be prepared for when things go awry.  The last Service Bus queue I created was the "myprocess"-error-queue.  Basically the message processing Logic App was wrapped in a Scope and upon error the message was sent to the error queue with an error message and error code. Also the error was used to create a service desk ticket request to our support team.  We currently do not have any automation in reprocessing the error, but as the use cases arise we will be implementing automation for the ones that we can.

This is the overview of how I used Azure Integration Services to work with an outside vendor and process messages regarding transactions occuring in their system.  I hope to dive deeper into each piece in future articles.  So stay on the lookout for future articles starting with Azure API Management.
